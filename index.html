DRF & FastAPI — From Planning to Production

Phase 1

Project Planning

1.1 Requirements Gathering

Before writing a single line, answer these questions thoroughly:

  • Who are the consumers of this API? Mobile apps, SPAs, third-party developers, internal microservices, or a combination?
  • What are the core business entities? Identify the nouns of your domain — users, orders, products, transactions, etc.
  • What operations must be supported? Map each entity to the actions users or systems need to perform on it — create, read, update, delete, search, aggregate, export.
  • What are the expected traffic patterns? Estimate concurrent users, requests per second, peak versus average load, and read-to-write ratios.
  • What are the non-functional requirements? Latency targets, uptime SLAs, compliance obligations (GDPR, HIPAA, PCI-DSS), data residency rules, and audit trail needs.
  • What integrations are required? Payment gateways, email providers, file storage services, third-party APIs, message brokers, analytics platforms.
  • What is the timeline and team composition? This directly influences technology choice, architecture complexity, and scope.

1.2 Technology Selection: DRF vs. FastAPI

This is the single most consequential early decision. Neither framework is universally better; the right choice depends on context.

🟢 Choose Django REST Framework when:
  • Your project needs a full-featured web framework beyond just the API layer — admin panels, template rendering, form handling, session management
  • You are working with a relational-heavy data model with complex relationships, and you want a mature, battle-tested ORM out of the box
  • Your team has strong Django experience and can leverage the enormous ecosystem of Django packages — django-allauth, django-celery-beat, django-filter, django-guardian, and hundreds more
  • You need a built-in admin interface for internal users or content management, which Django provides essentially for free
  • The project is a monolithic application where having everything in one cohesive framework reduces integration overhead
  • You value convention over configuration and want the framework to make many structural decisions for you
  • Your application is primarily synchronous in nature, with I/O-bound async needs handled by task queues
  • You need mature, well-documented authentication systems like session auth, token auth, OAuth2 via third-party packages
  • Long-term maintainability is paramount — 20-year track record, predictable release cycles, enormous community
🔵 Choose FastAPI when:
  • Your project is a dedicated API service or microservice that does not need an admin panel, template rendering, or the full Django ecosystem
  • High-concurrency I/O-bound workloads are central — many simultaneous connections to external APIs, databases, WebSockets, or streaming endpoints
  • You want native async/await support as a first-class citizen rather than an afterthought
  • Automatic OpenAPI documentation generated from type hints is important for developer experience and client SDK generation
  • Your team is comfortable with Python type hints and Pydantic for data validation
  • You are building microservices and want a lightweight, minimal framework that starts fast and consumes fewer resources
  • Performance benchmarks matter significantly — FastAPI on ASGI servers consistently outperforms DRF on WSGI for I/O-bound tasks
  • You want modern Python patterns — dependency injection, type-driven development, functional composition
  • The project needs WebSocket support, server-sent events, or streaming responses as core features
💡 Hybrid Consideration

It is entirely valid to use both within a single organization — Django with DRF for the primary application with its admin and ORM, and FastAPI for high-performance microservices that handle specific tasks like real-time notifications, data ingestion pipelines, or ML model serving.

1.3 Architecture Design Decisions

Monolith vs. Microservices

  • Start monolithic unless you have a clear reason not to. Premature microservice adoption introduces distributed systems complexity that small teams cannot afford.
  • DRF naturally fits the monolithic approach due to Django's "batteries included" philosophy.
  • FastAPI naturally fits the microservice approach due to its lightweight nature, but works perfectly as a monolith too.
  • Plan for a modular monolith — structure with clear boundaries, making future extraction into services possible without a rewrite.

Synchronous vs. Asynchronous

  • DRF is historically synchronous, running on WSGI servers. Django 4.1+ added async view support, but the ecosystem is still primarily synchronous. Offload long-running tasks to Celery, Django-Q, or Huey.
  • FastAPI is natively asynchronous, running on ASGI servers. You can mix sync and async endpoints freely.
  • Choose async when you have many concurrent connections waiting on I/O — external API calls, database queries on async-capable drivers, WebSocket connections.
  • Choose sync when your workload is CPU-bound, your database driver is synchronous, or your team is less experienced with async pitfalls.

API Style

  • Both frameworks are built for REST APIs. For GraphQL, consider Strawberry (both) or Graphene-Django (DRF). For gRPC, use grpcio with either framework.
  • Consider JSON:API or HAL for hypermedia-driven APIs if you need standardized pagination, filtering, and relationship representations.

1.4 Project Scoping

  • Define an MVP scope — the smallest useful API surface that delivers value. Resist the urge to design for every future possibility.
  • Create an API contract early using OpenAPI specification files as the contract between frontend and backend teams.
  • Identify bounded contexts — groups of related entities and operations that form cohesive units.
  • Establish versioning strategy from day one — URI versioning, header versioning, or query parameter versioning. Changing this later is painful.
Phase 2

Project Setup

2.1 Python Environment Management

  1. Choose a Python version: Use the latest stable release (3.11 or 3.12 recommended) for performance improvements and modern type hint support.
  2. Select an environment and dependency management tool:
    • pip + venv: Standard library approach. Simple, universally understood.
    • Poetry: Dependency resolution, virtual environment management, and packaging in one tool. Uses pyproject.toml.
    • PDM or Hatch: Modern alternatives following PEP 621 standards.
    • uv: Very fast, Rust-based drop-in replacement for pip and venv. Increasingly popular.
  3. Pin all dependencies including transitive ones for production builds.
  4. Separate dependency groups: Maintain distinct groups for production, development, and testing.

2.2 Version Control Setup

  1. Initialize a Git repository immediately — even before installing dependencies.
  2. Create a comprehensive .gitignore covering Python bytecode, virtual environments, IDE configurations, environment variable files.
  3. Establish a branching strategy:
    • Trunk-based development for small teams — short-lived feature branches merged frequently.
    • Git Flow for larger teams or projects with formal release cycles.
  4. Set up commit conventions: Use conventional commits (feat, fix, chore, docs, refactor, test).
  5. Configure pre-commit hooks for automated linting, formatting, type checking, and secret detection.
  6. Never commit secrets — no API keys, database passwords, or private certificates in the repository.

2.3 Project Structure

DRF Project Structure
  • The project directory contains root configuration — settings module, root URL configuration, WSGI/ASGI entry point.
  • Each app represents a bounded context — users, orders, products. Contains models, serializers, views, URLs, admin, signals, and tests.
  • Common utilities go in a dedicated utils or core app — custom permissions, pagination, exception handlers, base models, reusable mixins.
  • The settings module should be split into base, development, staging, and production files.
  • Place Celery task definitions within each app to maintain domain cohesion.
FastAPI Project Structure
  • FastAPI does not enforce a structure — establish conventions early.
  • Organize by feature or domain — each gets a package with router, schemas, models, service layer, dependencies, and tests.
  • A core or common package contains shared configuration, database session management, security, exception handlers, middleware.
  • The configuration module loads environment variables using Pydantic's BaseSettings with type validation.
  • Use a src layout to avoid import ambiguity.
⚠️ Critical Principle for Both

Separate concerns into layers: routes/views handle HTTP concerns, services contain business logic, repositories/ORM models handle data access, and schemas/serializers handle validation and transformation. Never put business logic directly in views or serializers.

2.4 Tooling Configuration

  • Linting: Use Ruff as the primary linter — replaces flake8, isort, pyupgrade with a single, extremely fast tool.
  • Formatting: Use Ruff's formatter or Black for consistent, opinionated formatting. Eliminate formatting debates from code reviews.
  • Type Checking: Use mypy or pyright. FastAPI projects benefit enormously from strict type checking.
  • Security Scanning: Configure Bandit for static analysis and pip-audit for dependency vulnerability scanning.
  • All tool configurations belong in pyproject.toml to avoid config file sprawl.
Phase 3

Database Design

3.1 Database Selection

PostgreSQL (Recommended Default)

  • Best general-purpose choice for both DRF and FastAPI. Excellent JSON fields, full-text search, array types, advanced indexing.
  • Django's ORM has the deepest PostgreSQL integration — ArrayField, JSONField, HStoreField, range fields, full-text search lookups.
  • FastAPI with SQLAlchemy also has excellent PostgreSQL support through asyncpg (async) or psycopg3 (sync/async).
  • Use PostGIS extension for geospatial data requirements.

MySQL/MariaDB

  • Viable but fewer advanced features. Choose only with existing MySQL expertise and infrastructure.
  • Be aware of character set and collation pitfalls, particularly utf8 vs utf8mb4.

SQLite

  • Development and testing only — never for production. Lacks concurrent write support.

MongoDB

  • Consider only if data is genuinely schema-less. Most applications that think they need MongoDB actually need PostgreSQL with a JSON column.
  • DRF: djongo or mongoengine (less mature). FastAPI: Motor (async) with Beanie or ODMantic.

Redis

  • Not a primary database but essential as caching layer, session store, message broker, and rate limiting backend.

3.2 Schema Design Principles

  • Normalize first, denormalize deliberately. Start with Third Normal Form and only denormalize with measured performance data.
  • Every table gets a primary key. Use UUIDs for public-facing identifiers (prevent enumeration attacks). Use auto-incrementing integers for internal PKs (join performance). You can have both.
  • Use created_at and updated_at timestamps on every table. Invaluable for debugging, auditing, and sync.
  • Implement soft deletes carefully. A deleted_at field allows recovery but complicates every query. Consider archive tables instead.
  • Design for the queries you will run, not just the data you will store. Examine API endpoints and identify access patterns.
  • Foreign keys should always have explicit ON DELETE behavior — CASCADE, SET NULL, PROTECT, or RESTRICT. Default to PROTECT.

3.3 Data Modeling Approaches

DRF Approach (Django ORM)
  • Models are Python classes inheriting from django.db.models.Model.
  • Use abstract base models for shared fields like timestamps, soft delete flags, or audit fields.
  • Model managers and QuerySet methods encapsulate common queries — keep query logic out of views and serializers.
  • Prefer abstract models over multi-table inheritance (avoid implicit JOIN overhead).
  • Model-level validation ensures data integrity regardless of entry point (API, admin, shell, management commands).
FastAPI Approach (SQLAlchemy / Tortoise ORM / SQLModel)
  • SQLAlchemy 2.0 is the most mature option. Use async extension with asyncpg for async database access.
  • SQLModel bridges SQLAlchemy and Pydantic, allowing one class for both. Convenient but can create coupling — use with caution for complex projects.
  • Tortoise ORM is Django-inspired and async. Simpler than SQLAlchemy but less powerful.
  • Maintain clear separation between database models and Pydantic schemas. They serve different purposes and will diverge.

3.4 Migration Strategy

DRF — Django Migrations
  • Django's migration system is automatic and robust. makemigrations detects changes, migrate applies them.
  • Always review generated migration files before committing.
  • Squash migrations periodically for stable apps with long migration chains.
  • Create separate data migration files using RunPython. Keep them idempotent and reversible.
  • Never edit a migration that has been applied to shared environments. Create a new one instead.
FastAPI — Alembic / Aerich
  • Alembic is the standard migration tool for SQLAlchemy. Requires more manual configuration than Django's system.
  • Auto-generated scripts require more review and manual adjustment than Django's.
  • Aerich is the migration tool for Tortoise ORM, with a similar workflow to Django's.
  • Test migrations both forward and backward.
📝 Shared Best Practices

Run migrations as a separate deployment step before starting new application instances. For zero-downtime deployments, make all schema changes backward-compatible: add columns as nullable first, deploy code that writes to both, then make non-nullable in subsequent deployments.

Phase 4

API Development

4.1 RESTful API Design Principles

  • Resources are nouns, not verbs. Use /orders, not /getOrders. The HTTP method conveys the action.
  • Use plural nouns for collection endpoints: /users, /products, /orders.
  • Nest resources to show relationships, but limit nesting to one level.

HTTP Methods

  • GET — retrieves resources. Must be safe and idempotent.
  • POST — creates new resources. Returns 201 Created with Location header.
  • PUT — replaces an entire resource. Idempotent.
  • PATCH — partially updates. Send only changed fields.
  • DELETE — removes a resource. Returns 204 No Content.

HTTP Status Codes

  • 200 — successful GET, PUT, PATCH
  • 201 — successful POST creating a resource
  • 204 — successful DELETE or no-body updates
  • 400 — client errors, validation failures
  • 401 — authentication failure (not identified)
  • 403 — authorization failure (identified but not permitted)
  • 404 — resource not found
  • 409 — conflicts (duplicate entries, state conflicts)
  • 422 — well-formed but semantically invalid (FastAPI's default)
  • 429 — rate limiting
  • 500 — unexpected server errors
Best Practice

Standardize error response format across all endpoints. Include a machine-readable error code, a human-readable message, and field-level error details for validation failures.

4.2 Endpoint Design

Collection Endpoints

  • Pagination is mandatory for all list endpoints:
    • Offset/limit: Simple, allows page jumping, degrades with large offsets.
    • Cursor-based: Performant at any depth, only forward/backward navigation. Best for infinite scroll or large datasets.
    • Page number: Familiar but has the same large-offset problem.
  • Filtering uses query parameters. DRF: django-filter. FastAPI: fastapi-filter or custom dependencies.
  • Sorting uses an ordering query parameter.
  • Search across multiple fields. DRF: built-in SearchFilter. FastAPI: manual implementation with PostgreSQL full-text search.

Action & Bulk Endpoints

  • For non-CRUD operations, use sub-resource endpoints: POST /orders/123/cancel
  • Bulk endpoints should return detailed per-item results indicating successes and failures.

4.3 Request & Response Handling

DRF Approach
  • Serializers are the core — validate incoming data, transform model instances to JSON.
  • ModelSerializer auto-generates fields from a model. Override for custom behavior.
  • Use separate serializers for read and write when request/response shapes differ.
  • ViewSets combine related views (list, create, retrieve, update, destroy). Routers auto-generate URL patterns.
  • Use APIView or generic views for more control or non-CRUD operations.
FastAPI Approach
  • Pydantic models (schemas) handle all validation and serialization. Define separate schemas for requests, responses, queries, and DB representations.
  • FastAPI automatically validates request data and returns 422 errors with detailed messages.
  • Use dependency injection for shared logic — DB sessions, current user, pagination, permissions.
  • Path operation functions are simpler than DRF views — receive validated typed params, return data.
  • For complex logic, call service functions from path operations.

4.4 Authentication & Authorization

Authentication Mechanisms

  • JWT Token-based (most common for SPAs/mobile):
    • DRF: djangorestframework-simplejwt
    • FastAPI: python-jose or PyJWT with dependency injection
    • Store access tokens in memory, refresh tokens in HTTP-only secure cookies
    • Access token: 5-15 min. Refresh token: days to weeks with rotation.
  • Session-based: Appropriate for same-origin web applications. Django has built-in support.
  • API key: Suitable for server-to-server communication. Hash keys before storing.
  • OAuth2: DRF uses django-oauth-toolkit. FastAPI uses built-in security schemes with authlib.

Authorization Patterns

  • Object-level permissions: DRF has permission classes. FastAPI uses dependency injection with HTTPException(403).
  • Role-based access control (RBAC): DRF leverages Django's groups/permissions system. FastAPI implements roles via user model fields or tables.
  • Attribute-based access control (ABAC): More flexible than RBAC but more complex.
  • Principle of least privilege: Default to denying access. Explicitly grant permissions.

4.5 API Versioning

  • URI versioning (/api/v1/users): Most explicit, easiest to understand.
  • Header versioning: Cleaner URLs but harder for clients.
  • Query parameter versioning: Easy to test but semantically awkward.
  • DRF has built-in versioning support for all approaches. FastAPI requires manual implementation.
  • Maintain backward compatibility within a version. Adding fields is safe; removing requires a new version.
Phase 5

Code Organization

5.1 Layered Architecture

Presentation Layer (Routes/Views)

  • Handles HTTP concerns only: parsing requests, returning responses, setting status codes.
  • Contains no business logic. If you find conditional logic beyond "which service to call," it's in the wrong place.
  • DRF: ViewSets, APIViews, generic views. FastAPI: path operation functions in router modules.

Service Layer (Business Logic)

  • Contains all business rules and orchestration. Enforces invariants, coordinates multi-step operations.
  • Services are framework-agnostic — should work from API endpoints, management commands, Celery tasks, or tests.
  • Each service handles one business operation: create_order, cancel_subscription, process_payment.

Data Access Layer (Repositories/ORM)

  • Encapsulates all database queries. In Django: model managers and QuerySets. In SQLAlchemy: repository classes.
  • Handles query optimization — select_related, prefetch_related, deferred loading.

Schema/Serializer Layer

  • Defines data shapes for API requests and responses.
  • Multiple schemas per entity are normal — creation, update, list, detail, internal.

5.2 Settings & Configuration Management

DRF — Django Settings
  • Split into base.py, development.py, staging.py, production.py within a settings package.
  • Load secrets from environment variables using django-environ or python-decouple.
  • Never use DEBUG=True in production.
FastAPI — Pydantic Settings
  • Use Pydantic's BaseSettings for typed, validated configuration. Environment variables loaded automatically.
  • Support .env files for local development.
  • Use @lru_cache on the settings factory to load once and reuse.
  • Group into nested models — DatabaseSettings, RedisSettings, AuthSettings.
📝 Twelve-Factor Compliance

Store configuration in the environment. Maintain a .env.example file. Validate configuration at startup — fail fast with clear errors rather than failing at runtime.

5.3 Dependency Injection

  • DRF: Less formal — uses settings, middleware, signals, mixins, and serializer context for dependency management.
  • FastAPI: First-class Depends() system — central to the framework. Dependencies can be sync/async functions, classes, or generators. Supports recursive resolution, testing overrides, and cross-cutting concerns (auth, DB sessions, rate limiting).
Phase 6

Testing

6.1 Testing Philosophy

  • Tests are not optional. Code without tests is a liability.
  • Aim for a testing pyramid: many unit tests (fast), fewer integration tests, small number of end-to-end tests.
  • Test behavior, not implementation. Makes tests resilient to refactoring.
  • Write tests that serve as documentation — developers should understand behavior by reading tests.

6.2 Strategy by Layer

Unit Tests

  • Test service layer functions in isolation. Mock database and external calls.
  • Test serializers/schemas with various valid and invalid inputs.
  • Should run in milliseconds. If slow, you're hitting real infrastructure — mock it.

Integration Tests

  • Test API endpoints with HTTP requests. Verify status codes, headers, response body.
  • DRF: APIClient. FastAPI: TestClient (sync) or httpx.AsyncClient (async).
  • Test authentication and authorization — unauthenticated rejection, resource ownership, role restrictions.
  • Use a real test database (not mocks) for integration tests.

End-to-End Tests

  • Test complete workflows spanning multiple endpoints.
  • Slow and brittle — minimize number, focus on critical business workflows.

6.3 Test Infrastructure

Test Data Management

  • Factories over fixtures. Use factory_boy (Django) or factory patterns (FastAPI). Composable, maintainable, explicit.
  • Faker for realistic random data. Seed for reproducible runs.
  • Isolate test data — each test creates its own data, no dependencies on other tests.
DRF-Specific Testing
  • TestCase wraps each test in a transaction that rolls back, ensuring isolation.
  • @override_settings for testing under different configurations.
  • SimpleTestCase for tests without database access (faster).
FastAPI-Specific Testing
  • Use pytest with pytest-asyncio for async endpoints and services.
  • Override dependencies via app.dependency_overrides for mocks and test DBs.
  • Use HTTPX's async client to test actual async execution paths.

6.4 Test Quality Metrics

  • Code coverage: Aim 80-90% as guideline, but 100% doesn't guarantee correct tests.
  • Focus coverage on business-critical paths — payments, auth, validation, permissions.
  • Use mutation testing (mutmut) to verify tests actually catch bugs.
  • Run tests in CI on every commit. Failing tests block merges. No exceptions.
Phase 7

Performance & Optimization

7.1 Database Query Optimization

The N+1 Query Problem

  • The most common performance killer in both frameworks.
  • DRF/Django: Use select_related (JOINs for FK/OneToOne) and prefetch_related (separate query + Python join for M2M/reverse FK).
  • FastAPI/SQLAlchemy: Use joinedload or selectinload in queries.
  • Monitor queries in development. Use django-debug-toolbar or SQLAlchemy echo mode. 100+ queries on a list endpoint = N+1 problem.

Indexing Strategy

  • Index every column used in WHERE, ORDER BY, and JOIN clauses.
  • Composite indexes: Column order matters. Most selective column first.
  • Partial indexes for commonly filtered subsets.
  • Don't over-index. Each index slows writes and consumes storage.
  • Use EXPLAIN ANALYZE in PostgreSQL for slow query analysis.

Additional Techniques

  • Use database-level aggregation instead of fetching all rows and computing in Python.
  • Paginate aggressively. Never return unbounded result sets.
  • Use only()/defer() (Django) or load_only() (SQLAlchemy) to fetch only needed columns.
  • Use database connection pooling — PgBouncer or built-in SQLAlchemy pooling.

7.2 Caching Strategy

Cache Layers

  1. Application-level: Redis or Memcached for expensive queries, computed values.
  2. HTTP-level: Cache-Control headers, ETag, Last-Modified.
  3. CDN caching: Static assets and public API responses.

Caching Patterns

  • Cache-aside (lazy loading): Check cache → miss → fetch from DB → store → return. Most common.
  • Write-through: Update cache on every DB write. Always current but adds write latency.
  • Write-behind: Update cache immediately, persist to DB asynchronously. Best performance, risks data loss.
  • Cache invalidation is the hardest problem. Use TTL as simplest strategy, event-based for critical data.
  • DRF: Django cache framework + django-cacheops or django-cachalot for automatic ORM-level caching.
  • FastAPI: Redis directly via async client + fastapi-cache2 for decorator-based caching.

7.3 Asynchronous Processing

Background Tasks

  • Never perform long-running operations in request handlers. Emails, image processing, reports, slow API calls go to background tasks.
  • DRF/Django: Celery (standard, with Redis/RabbitMQ). Django-Q2 as simpler alternative.
  • FastAPI: Celery for robust processing. Built-in BackgroundTasks for lightweight, non-critical tasks. ARQ for async-native task queue.

Task Design

  • Make tasks idempotent — same task twice = same result. Essential for safe retries.
  • Set timeouts and retry limits for all tasks.
  • Monitor task queues — depth, processing time, failure rates. Use Flower for Celery.

7.4 Performance Monitoring

  • Instrument with APM tools — New Relic, Datadog, Sentry Performance, or OpenTelemetry.
  • Track endpoint response times at p50, p95, p99. Averages hide tail latency.
  • Monitor database query count and duration per request.
  • Profile memory and CPU. Watch for memory leaks in long-running workers.
  • Load test before launch using Locust or k6 at 2-3x expected peak traffic.
Phase 8

Security Implementation

8.1 OWASP Top 10 Protection

Injection Attacks

  • Always use parameterized queries. Both Django ORM and SQLAlchemy do this by default. Never use string concatenation for SQL.
  • Validate and sanitize all input from every source.

Broken Authentication

  • Hash passwords using bcrypt, argon2, or scrypt — never MD5 or SHA-256.
  • Enforce strong password policies and check against breached password lists.
  • Implement account lockout after repeated failed attempts.

Broken Access Control

  • Check permissions on every request, not just initial authentication.
  • Validate resource ownership — verify order 123 belongs to the requesting user.
  • Use UUIDs instead of sequential IDs for public resource identifiers.

Security Misconfiguration

  • Disable debug mode in production.
  • Set security headers: CSP, X-Content-Type-Options, X-Frame-Options, HSTS, X-XSS-Protection.
  • Restrict CORS origins. Never use wildcard (*) for authenticated APIs.

Sensitive Data Exposure

  • Use HTTPS everywhere. Configure HSTS headers.
  • Never log sensitive data — passwords, tokens, credit cards, PII.
  • Encrypt sensitive data at rest and mask in API responses.

8.2 Rate Limiting

  • Implement on all endpoints, stricter on auth endpoints.
  • DRF: Built-in throttling classes (AnonRateThrottle, UserRateThrottle, ScopedRateThrottle).
  • FastAPI: Use slowapi with Redis backend.
  • Use sliding window rate limiting. Return 429 with Retry-After header.

8.3 Input Validation

  • Validate everything: types, lengths, ranges, formats, file types and sizes.
  • Whitelist acceptable values rather than blacklisting dangerous ones.
  • Validate at schema level for format and service level for business rules.
  • Limit request body size at server/middleware level.
  • Sanitize file uploads — validate MIME by content inspection, scan for malware, store in isolated locations.

8.4 Secrets Management

  • Never store secrets in source code, committed config files, or Docker images.
  • Use environment variables as minimum viable approach.
  • For production: HashiCorp Vault, AWS Secrets Manager, Google Secret Manager, or Doppler.
  • Rotate secrets regularly. Use different secrets per environment.
Phase 9

Documentation

9.1 API Documentation

DRF Documentation
  • drf-spectacular is recommended for generating OpenAPI 3.0 schemas. Introspects serializers, views, and URLs.
  • drf-yasg is an older alternative (Swagger 2.0). Use drf-spectacular for new projects.
  • Both provide Swagger UI and ReDoc interfaces.
  • Annotate views and serializers with docstrings and decorators for descriptions and examples.
FastAPI Documentation
  • Generates OpenAPI documentation automatically from type hints and Pydantic models — one of FastAPI's strongest features.
  • Swagger UI at /docs and ReDoc at /redoc with zero configuration.
  • Type hints drive documentation accuracy. Use descriptive field names, Field() annotations, and Enum types.
  • Add tags, summaries, descriptions, and response models to enrich docs.
📝 Shared Practices

Keep docs in sync with code by generating from code. Include request/response examples for every endpoint. Document authentication requirements and error responses with status codes and body formats. Version documentation alongside API versions.

9.2 Internal Documentation

  • Maintain Architecture Decision Records (ADRs) for significant technical decisions and their reasoning.
  • Write a comprehensive README — purpose, quick start, environment setup, common tasks, testing, deployment.
  • Document environment variables with purpose, format, defaults, and required/optional status.
  • Create runbooks for operational procedures — migrations, backups, incident response, rollbacks, scaling.
Phase 10

Deployment

10.1 Application Server Selection

DRF Server
  • Gunicorn is the standard WSGI server. Workers = (2 × CPU cores) + 1.
  • Use Gunicorn with Uvicorn workers for async support.
  • Never use Django's runserver in production.
FastAPI Server
  • Uvicorn is the primary ASGI server. Use --workers flag or Gunicorn with UvicornWorker.
  • Hypercorn is an alternative with HTTP/2 support.

Reverse Proxy

  • Always place Nginx, Caddy, or a cloud load balancer in front of the application server.
  • Handles TLS termination, static files, request buffering, connection management, and security filtering.
  • Configure health check endpoints for availability monitoring.

10.2 Containerization

  • Use multi-stage Docker builds: build stage with dev deps, runtime stage with production only.
  • Base image: python:3.12-slim (safer default) or python:3.12-alpine (smaller but C compilation issues).
  • Run as non-root user inside the container.
  • Docker Compose for local dev only. Use Kubernetes, ECS, or managed platforms for production.
  • Pin all versions — base image tags and pip packages for reproducible builds.
  • Use .dockerignore to exclude VCS, venvs, local config, test data from build context.

10.3 CI/CD Pipeline

Continuous Integration

  1. Trigger on every push to any branch and every PR to main.
  2. Lint and format check — fail on violations.
  3. Type check — fail on type errors.
  4. Run full test suite against real PostgreSQL (not SQLite).
  5. Measure coverage — fail if below threshold.
  6. Security scan — dependency vulnerabilities and static analysis.
  7. Build Docker image to verify build success.

Continuous Deployment

  1. Build and tag Docker image with git commit hash / semantic version.
  2. Push to container registry (ECR, GCR, GHCR).
  3. Run migrations separately — not as part of app startup.
  4. Deploy to staging automatically on merge to main. Run smoke tests.
  5. Deploy to production with manual approval for high-risk changes.
  6. Use blue-green or rolling deployments for zero downtime.
  7. Canary deployments for high-risk changes — small traffic %, monitor, gradually increase.
  8. Automated rollback if health checks fail.

CI/CD Tools

  • GitHub Actions: Tight GitHub integration, extensive marketplace.
  • GitLab CI/CD: Built into GitLab.
  • Jenkins: Self-hosted, highly customizable, requires maintenance.
  • CircleCI, Buildkite: Third-party with various strengths.

10.4 Infrastructure Decisions

  • AWS: Most comprehensive. EC2/ECS, RDS, ElastiCache, S3, CloudFront, SQS.
  • GCP: Strong in Kubernetes (GKE) and ML. Cloud Run for serverless containers.
  • Azure: Strong enterprise integrations.
  • DigitalOcean, Render, Railway, Fly.io: Simpler platforms for smaller projects.
  • Use managed services for databases, caching, message brokers unless you have a compelling reason to self-host.
  • For most projects, a managed container platform (ECS, Cloud Run) is sufficient over Kubernetes.

Phase 11

Maintenance & Scaling

11.1 Logging

Logging Strategy

  • Use structured logging (JSON) — searchable, parseable, integrable with aggregation tools.
  • Log levels have meaning:
    • DEBUG: Detailed diagnostics, disabled in production.
    • INFO: Significant normal events — user registered, order placed.
    • WARNING: Unexpected but recoverable — deprecated API used, slow query.
    • ERROR: Failures preventing operation — payment failed, DB connection lost.
    • CRITICAL: System-level failures requiring immediate attention.
  • Include context: request ID, user ID, endpoint, timestamp, duration, business identifiers.
  • Request ID tracing: Generate unique ID at entry point, include in every log entry and outbound call.

Log Aggregation

  • Send to centralized platform: ELK Stack, Grafana Loki, Datadog, CloudWatch.
  • Set up log-based alerts for error rate spikes and security events.
  • Retain logs: 30-90 days operational, 1-7 years audit.

11.2 Monitoring & Alerting

Application Metrics

  • Response time distribution — p50, p95, p99 per endpoint. Alert on p95 degradation.
  • Error rate — percentage of 5xx responses.
  • Request throughput — requests per second over time.
  • DB connection pool utilization and task queue depth.

Health Checks

  • Liveness endpoint (/healthz): Returns 200 if process is running. Used by orchestrators to restart unhealthy instances.
  • Readiness endpoint (/ready): Verifies DB, cache, and dependency connectivity. Used to route traffic only to ready instances.

Alerting Principles

  • Alert on symptoms, not causes. Alert on "response time > 500ms" not "CPU > 80%."
  • Reduce alert noise. Every alert should require action.
  • Use escalation policies — on-call → team lead → engineering management.

11.3 Scaling Strategies

Vertical Scaling

  • Increase resources on existing servers — simple but has limits.
  • Scale the database vertically first. More RAM for query caching is often the most impactful action.

Horizontal Scaling

  • Add more application instances behind a load balancer. Requires stateless application — no in-memory sessions, no local file storage.
  • Session storage in Redis, file storage in S3/GCS.
  • Load balancing: Round-robin, least-connections, or IP-hash (avoid if possible).

Database Scaling

  • Read replicas for read-heavy workloads. DRF: Django database routers. FastAPI: separate connection pools.
  • Connection pooling with PgBouncer or built-in configs.
  • Table partitioning for very large tables.
  • Sharding as last resort — enormous complexity, avoid unless all other options exhausted.

Async Processing Scaling

  • Scale Celery workers independently from the web application.
  • Use separate queues for different priorities.
  • Auto-scale workers based on queue depth using Kubernetes HPA or cloud auto-scaling.

11.4 Long-Term Maintenance

  • Update dependencies regularly — monthly for minor versions, weekly for security patches. Use Dependabot or Renovate.
  • Plan major framework upgrades as deliberate projects. Read release notes, identify breaking changes, update incrementally.
  • Track technical debt explicitly — create tickets, allocate sprint time.
  • Refactor continuously in small increments rather than planning large rewrites.
  • Automated database backups with retention policies. Test restores regularly.
  • Data archival strategy for old records — cold storage or archive tables.
  • GDPR/privacy compliance: Data deletion, export endpoints, consent management.
  • Document tribal knowledge — ADRs, wiki pages, inline comments for non-obvious context.
  • Onboarding documentation for new team members.

Final Reminders

Core Principles to Remember

🎯
Start simple, measure, then optimize. Premature optimization wastes time on problems that may not exist.
🤖
Automate everything you do more than twice — testing, deployment, database migrations, environment setup.
🔒
Security is not a phase; it is a constant. Build it into every layer from the beginning.
🤝
Your API is a contract. Breaking changes break trust. Version deliberately and communicate proactively.
👁️
Logs, metrics, and alerts are not optional. You cannot fix what you cannot see.
💾
Back up your data, and test your restores. A backup that has never been restored is not a backup, it is a hope.
🧪
Write tests that matter. Critical paths first, edge cases second, happy paths last.
📝
Document your decisions, not just your code. Code shows what and how; documentation explains why.
🏗️
Choose boring technology when possible. PostgreSQL, Redis, Celery, Nginx — battle-tested and reliable. Save your complexity budget for your domain.
